Avoiding Rerendering – Friday, Nov. 19, 2021

By nature, react components rerender automatically whenever a change occurs in its state or props. However, unnecessary rerenders of components can slow down performance, which is always no bueno. To learn how to avoid this inefficiency, I turned to this tutorial by DebugBear. The idea is to only update components when needed, rather than automatically, and also to avoid unintentional rerenders.

One issue is that whenever a parent component rerenders, its children components rerender, too. To avoid this, the React.memo component can be used. According to the React documentation, React.memo is a higher order component that memoizes results of a call, meaning that rather than rerendering a component, the last rendered result is used instead. In short, React.memo only rerenders when props change. So if we have a Counter component, we would say: 

const Counter = React.memo(function Counter({ name, value }) {
    console.log(`Rendering counter ${name}`);
    return (
        <div>
            {name}: {value}
        </div>
    );
});

So that if we had multiple counters, only the one whose value has changed would be rerendered. Preventing rerendering becomes a little more complicated when props like onClickIncrement are passed, however; the prop changes from every rerender. To solve this, we can use a useCallback hook instead to memoize. This is what that would look like: 

<Counter
    name="A"
    value={counterA}
    onClickIncrement={React.useCallback(() => setCounterA(counterA + 1), [counterA])}
/>

So now when counter A changes value, it is the only counter to rerender. This approach tackles the issue of unintentional prop changes triggering component rerenders. Another issue occurs with passing object literals as props, because every time App renders, a new object must be created. A quick solution to this is, when applicable, simply creating the object once outside of the App component. This works in the case that the object stays the same, like a header. For dynamic objects, we can once again use a useMemo hook. In this case, the header component would only rerender if, say, the font was changed, rather than every time a counter changes.

Overall, it seems like there's a lot of room for inefficiency-- this blog barely scracthed the surface of possible rerendering scenarios. There's probably a fine line between excessive optimization and functionally working code.
